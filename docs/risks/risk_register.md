# Реестр рисков проекта «Выгодный Взгляд»

## Шкала оценки
- **Вероятность:** 1 (очень низкая) – 5 (очень высокая)
- **Влияние:** 1 (незначительное) – 5 (критическое)

| № | Риск | Описание | Вероятность | Влияние | Меры предотвращения | План реагирования (если риск наступил) |
|---|------|----------|-------------|---------|----------------------|----------------------------------------|
| 1 | **Блокировка парсинга маркетплейсами** | Ozon или WB распознают бота и блокируют IP или возвращают капчу | 4 | 4 | Ротация User-Agent, задержки (1-3 сек), использование прокси-пулов, кеширование результатов | Переключиться на резервные прокси, уменьшить частоту запросов, уведомить команду, рассмотреть переход на официальное API |
| 2 | **Отсутствие официального API у WB** | Wildberries не предоставляет открытого API для поиска товаров по запросу | 5 | 3 | На MVP использовать парсинг HTML с заглушкой для доставки. Изучить недокументированные API | Если парсинг полностью заблокирован – перейти только на Ozon и Яндекс.Маркет, расширить позже |
| 3 | **Низкая точность сопоставления товаров** | Один и тот же товар у разных продавцов отображается как разные позиции (дубликаты) | 3 | 3 | Разработать эвристики группировки по названию и артикулу (например, сравнение строк) | Вручную пометить дубликаты в админке, доработать алгоритм группировки |
| 4 | **Несоблюдение дедлайнов командой** | Кто-то из участников не сдаёт вовремя отчёты или задачи | 3 | 4 | Ежедневные чат-отчёты (daily), прозрачное распределение задач, напоминания за 2 дня до дедлайна | Связаться с участником, выяснить причины, перераспределить задачи, предупредить куратора |
| 5 | **Расхождение дизайна и вёрстки** | Разработчики реализуют интерфейс не так, как задумано дизайнером | 3 | 2 | Ранняя передача UI-кита, регулярные синхронизации дизайнера с разработчиками, использование Figma-инспектора | Провести дополнительный созвон, скорректировать вёрстку, при необходимости переделать часть компонентов |
| 6 | **Проблемы с хранением истории цен** | Рост объёма данных при отслеживании цен на множество товаров | 2 | 3 | Использовать агрегацию (хранить только изменения), ограничить глубину истории 90 днями, выбрать эффективную БД | Перейти на облачное хранилище, настроить архивацию старых данных |
| 7 | **Изменение приоритетов куратором** | Куратор в середине спринта просит добавить новый функционал | 2 | 3 | Чётко согласовать MVP на старте, все изменения – через запрос на изменение (Change Request) | Оценить трудозатраты, перенести часть текущих задач в следующий спринт, согласовать новый план |
| 8 | **Недостаточная производительность** | Поиск работает дольше 5-10 секунд, пользователи разочарованы | 2 | 4 | Использовать кеширование, параллельные запросы к маркетплейсам, асинхронную обработку | Оптимизировать парсеры, добавить индикатор загрузки, при необходимости масштабировать бэкенд |
